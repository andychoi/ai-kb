# /kb-cluster — Auto-MOC generation via Louvain

Manual-only slash command. Produces one Map-of-Content note per cluster of densely-interlinked notes in `notes/`, `refs/`, `sources/`, plus a `clusters/_index.md`. Every MOC file is **fully auto-generated** — edits are lost on next run (per the design at `docs/superpowers/specs/2026-05-16-kb-cluster-design.md`).

## Arguments

- `--min-size N` (default `3`) — drop clusters with fewer than N members.
- `--resolution R` (default `1.0`) — Louvain resolution; higher = more, smaller clusters.
- `--dry-run` — compute and print the plan to stdout, write no files, no commit.

## Pre-flight

1. **Verify the venv exists.** If `.venv-cluster/bin/python3` is missing → write to stderr:
   ```
   kb-cluster: .venv-cluster/ not found. Run `bin/cluster/setup-cluster.sh` first.
   ```
   and exit `1`.

2. **`git pull --rebase`** if a remote is configured. If conflicts, surface and bail (do not auto-resolve).

## Behavior

### Step 1 — Run the graph helper

```sh
.venv-cluster/bin/python3 bin/cluster/kb_cluster.py \
  --vault "$PWD" --min-size "${MIN_SIZE:-3}" --resolution "${RESOLUTION:-1.0}"
```

Capture stdout as JSON. The helper exits `1` with stderr describing frontmatter issues if any note in scope has bad frontmatter — in that case, surface the message and exit `1`. The user should run `/kb-validate --fix` first.

JSON shape:

```json
{
  "clusters": [
    {
      "canonical_id": "01HKZX...",
      "canonical_title": "Claude API",
      "slug": "claude-api",
      "size": 12,
      "members": [
        {"id": "01HKZX...", "title": "Claude API", "path": "refs/concepts/claude-api.md"},
        ...
      ]
    },
    ...
  ],
  "stats": {"nodes": 87, "edges": 142, "dropped": 3}
}
```

If `clusters` is empty: emit `kb-cluster: no clusters produced (size threshold or empty graph)` to stderr, exit `0`. No commit.

### Step 2 — Load persistent state

Read `.kb/clusters/state.json` (create if missing as `{"schema_version": 1, "clusters": {}}`).

For each cluster in the JSON:

- If `state.clusters[canonical_id]` exists → reuse its `moc_ulid`, `first_seen`, and `slug` (the persisted slug — even if the canonical_title shifted casing). This preserves git blame.
- Otherwise → mint a new ULID (see CLAUDE.md §5), set `first_seen` to this run's ISO-8601 UTC timestamp.

Always update `last_seen` to this run's timestamp.

For the `_index.md` itself: use the reserved key `"_index"` in state.json the same way (mint once, reuse forever).

### Step 3 — Generate subtitles

For each cluster, generate a one-line subtitle (~80 chars) summarizing its theme. Use the top 5–8 member titles and your judgement. This is *your* writing — no separate LLM client call.

Examples:

| Members (top) | Good subtitle |
|---|---|
| Claude API, Prompt caching, Tool use, SDK migration | `Anthropic SDK patterns: caching, tool use, version migration.` |
| watchfiles, file-watch daemon, inbox watch, Phase 2 | `Phase 2 file-watch daemon and its supporting concepts.` |

### Step 4 — Write the MOCs

For each cluster, write `clusters/<slug>.md` (overwrite if present) with:

```yaml
---
id: <moc_ulid>
type: note
title: "Cluster: <canonical_title>"
created: <first_seen>
updated: <this-run ISO-8601 UTC>
tags: [auto-cluster]
source: manual
idem_key: <moc_ulid>
subtitle: "<your subtitle from step 3>"
cluster_size: <size>
canonical_member: <canonical_id>
---

> Auto-generated by `/kb-cluster`. Manual edits will be lost on next run.

## Members (<size>)

- [[<member-1-title>]]
- [[<member-2-title>]]
- ...
```

Order members by the order the helper returned (already sorted by in-degree desc, then title).

### Step 5 — Write `clusters/_index.md`

```yaml
---
id: <_index ULID from state.json>
type: note
title: "Cluster index"
created: <_index first_seen>
updated: <this-run ISO-8601 UTC>
tags: [auto-cluster, index]
source: manual
idem_key: <id>
---

> Auto-generated by `/kb-cluster`. <N> clusters covering <M> notes; <D> small clusters dropped (min-size=<N>).

| Cluster | Size | Canonical member |
|---|---|---|
| [[Cluster: <canonical_title_1>]] | <size> | [[<canonical_title_1>]] |
| ... | ... | ... |
```

Rows sorted by `size` desc, then `slug`.

### Step 6 — Clean up stale MOCs

1. List `clusters/*.md` (exclude `_index.md` and `.gitkeep`).
2. Compute `produced = {slug for cluster in this run}`.
3. For each file not matching a produced slug → `git rm` it.
4. Remove the corresponding `state.clusters[canonical_id]` entries whose slug was deleted.

### Step 7 — Persist state and commit

1. Write the updated `.kb/clusters/state.json` (sorted keys, 2-space indent).

2. Stage: `clusters/` and `.kb/clusters/state.json` (do **not** use `git add -A`).

3. Commit with the §6 grammar:
   ```
   kb: cluster clusters/ (<N> clusters, <M> members)

   Co-Authored-By: Claude <noreply@anthropic.com>
   ```
   Use `kb-bot` identity only if `KB_BOT=1` env is set; otherwise the user's git identity (this is a manual command per CLAUDE.md §7).

4. If `--dry-run`: print the would-be commit message and the diff summary, write nothing, do not commit.

### Step 8 — Report

Emit to stdout:

```
kb-cluster: <N> clusters written, <D> small dropped (<nodes> nodes, <edges> edges)
  - <slug-1> (<size>): <subtitle>
  - <slug-2> (<size>): <subtitle>
  ...
```

Exit `0` on success.

## Headless contract

- Non-interactive. Missing args → defaults.
- Idempotent in `clusters/` content modulo timestamps and subtitle wording.
- Idempotent in `.kb/clusters/state.json` — same canonical IDs across runs reuse their MOC ULIDs.
- Exit `0` only on successful commit (or successful dry-run, or empty-clusters case).

## Failure modes

| Condition | Behavior |
|---|---|
| `.venv-cluster/` missing | stderr message, exit 1 |
| Frontmatter parse error in scope | stderr surfaces paths, exit 1 |
| No clusters produced | stderr message, exit 0, no commit |
| Git pull rebase conflict | surface conflict, exit 1, do not auto-resolve |
| Git commit fails | exit 1, do not retry |
